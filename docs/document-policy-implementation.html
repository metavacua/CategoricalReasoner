Purpose

This document provides operational implementation guidance for the formal model defined in Formal Document Policy: Categorical Reasoner Repository. It translates the mathematical definitions into actionable rules for contributors and automated tools.

Warning: Violations of these rules constitute formal inconsistencies that must be resolved before commits are accepted.

2. File Placement Rules

2.1. The Golden Rule

Golden Rule: With the sole exception of GitHub special files, ALL documentation must reside in the docs/ directory. 
FOR EACH file f:
    IF f IS documentation AND f IS NOT special:
        ASSERT Location(f) STARTS WITH "docs/"
    ELSE IF f IS documentation AND f IS special:
        ASSERT Location(f) == "/" (root)
    ELSE IF f IS software:
        ASSERT Location(f) STARTS WITH "src/"
    END IF
END FOR

2.2. Documentation Placement

Rule D-1: Explanatory prose, thesis text, and research exposition must be placed in docs/.
Content Type Correct Location Incorrect Location
Thesis chapters docs/dissertation/ /thesis.md
Mathematical definitions docs/structural-rules/ /math-defs.md
Standards reproductions docs/standards/ /standards.md
API documentation docs/api/ /api.md

2.3. Software Placement

Rule S-1: All executable code, source files, and programmatic artifacts must be placed in src/.
Content Type Correct Location Incorrect Location
Java source files src/main/java/ /MyClass.java
Unit tests src/tests/ /tests/
Benchmark queries src/benchmarks/queries/ /queries/
Build scripts src/scripts/ /build.sh

2.4. Special Files Placement

Rule SP-1: GitHub special files and tooling configuration files may exist at the repository root.
File Required Location Justification
README.md Root GitHub renders this on repository page
LICENSE Root GitHub displays license badge
AGENTS.md Root + each directory Repository-specific special file
.gitignore Root Git configuration requires root placement
pom.xml Root Maven requires project descriptor at root
.reuse/dep5 Root .reuse/ subdirectory REUSE specification requirement

2.5. LLM-Generated Content

Rule LLM-1: No arbitrary LLM-generated files are permitted in the repository root.

All AI-generated documentation must be:

  1. Curated and validated by a human
  2. Placed in the appropriate subdirectory under docs/ or src/
    3. Reviewed for compliance with the formal model
  3. Tagged with appropriate metadata (PROV-O records)
Bad: Committing /ai-generated-analysis.md to root
Good: Committing docs/dissertation/chapter4-analysis.html with proper semantic markup

3. License Header Requirements

3.1. REUSE Specification Compliance

All copyrightable files must include REUSE-compliant license headers as per REUSE Software Specification v3.0.

3.2. CC BY-SA v4.0 Header Pattern (Documentation)

SPDX-FileCopyrightText: 2025 Ian Douglas Lawrence Norman McLean

SPDX-License-Identifier: CC-BY-SA-4.0

This header applies to all files in the docs/ directory, including:

  • HTML/XHTML documentation files
  • Markdown exposition files (README.md in docs/)
  • AGENTS.md policy files
  • Standards reproductions (where applicable)

3.3. AGPL v3.0 Header Pattern (Software)

SPDX-FileCopyrightText: 2025 Ian Douglas Lawrence Norman McLean

SPDX-License-Identifier: AGPL-3.0-or-later

This header applies to all files in the src/ directory, including:

  • Java source files
  • Build scripts
  • Test files
  • Benchmark code
  • AGENTS.md files in src/ (which describe software policies)

3.4. Non-Copyrightable Files

The following files are explicitly non-copyrightable and must NOT include license headers:

  • LICENSE file (contains license text)
  • Verbatim reproductions of public domain works
  • Factual data compilations without creative arrangement

3.5. License Header Placement

File Type Header Location Format
Java Top of file, after package statement /* ... */ block comment
HTML In <head> as <meta> tags SPDX metadata
Markdown Top of file Comment-style with SPDX prefix
Shell/Bash Shebang line Inline comment with SPDX prefix

4. Validation Procedures

4.1. Automated Validation

The following automated checks must be configured as pre-commit hooks and CI/CD gates:

V-1: REUSE Compliance Check

$ reuse lint
# Must return exit code 0

V-2: File Placement Check

$ ./scripts/validate-file-placement.sh
# Must reject LLM-generated files in root
# Must enforce docs/ for documentation
# Must enforce src/ for software

V-3: AGENTS.md Consistency Check

$ ./scripts/validate-agents-md.sh
# Must verify AGENTS.md exists in each directory
# Must verify compliance with AGENTS.md standard

V-4: License Consistency Check

$ ./scripts/validate-licenses.sh
# Must verify docs/ files have CC-BY-SA-4.0
# Must verify src/ files have AGPL-3.0-or-later
# Must verify root special files have appropriate licenses

4.2. Manual Validation

Before committing, contributors must manually verify:

  1. All documentation is placed under docs/ (unless it's a GitHub special file)
  2. All software code is placed under src/
  3. LLM-generated content is not arbitrarily placed at repository root
  4. Appropriate license headers are present on all copyrightable files
  5. AGENTS.md files are present and up-to-date in modified directories

4.3. Disambiguation Procedure

When a file's classification is ambiguous:

  1. Check Format: Is it code or prose?
  2. Check Purpose: Is it explanatory or executable?
  3. Check Intent: Was it generated as documentation or software?
  4. Apply Heuristics:
    • If primarily human-readable exposition → docs/
    • If primarily machine-executable → src/
    • If metadata about documentation → docs/
    • If metadata about software → src/
  5. Document Decision: Add comment or AGENTS.md entry explaining rationale

 * A sequent in subclassical logic.
 *
 * This file is part of the CategoricalReasoner software and is

public record Sequent(List<Formula> antecedents, Formula succedent) {

    }
}
