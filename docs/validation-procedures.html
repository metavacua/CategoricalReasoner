Overview

This document defines validation procedures to ensure compliance with the formal document policy defined in Formal Document Policy and Implementation Guide.

Presumption of Inconsistency: All artifacts must be validated before commit. Failed validations constitute formal refutations requiring resolution.

1. Automated Validation

1.1. REUSE Specification Compliance

Check V-1: License Headers

$ reuse lint

Expected Result: Exit code 0, no violations reported

What it checks:

  • All copyrightable files have SPDX license headers
  • License headers match file type conventions
  • License identifiers are valid SPDX identifiers
  • License text files exist where required
Failure Mode: Missing or incorrect license headers

1.2. File Placement Validation

Check V-2: Root Directory Cleanliness

$ ./scripts/validate-root-files.sh

Expected Result: No arbitrary LLM-generated files in root

What it checks:

  • Only GitHub special files exist at repository root
  • No arbitrary documentation files (.md, .html, .txt) in root
  • No source code files (.java, .class) in root
  • Special files match allowed list

Check V-3: Documentation Placement

$ ./scripts/validate-docs-placement.sh

Expected Result: All documentation files in docs/

What it checks:

  • HTML/XHTML files only in docs/ (or subdirectories)
  • Explanatory Markdown only in docs/ (except special files)
  • Thesis content in docs/dissertation/
  • Standards reproductions in docs/standards/

Check V-4: Software Placement

$ ./scripts/validate-src-placement.sh

Expected Result: All code files in src/

What it checks:

  • Java source files only in src/
  • Test files in src/tests/
  • Benchmark queries in src/benchmarks/queries/
  • Build scripts in src/scripts/

1.3. AGENTS.md Compliance

Check V-5: AGENTS.md Presence

$ ./scripts/validate-agents-md.sh

Expected Result: AGENTS.md in every directory with content

What it checks:

  • AGENTS.md exists in root
  • AGENTS.md exists in docs/
  • AGENTS.md exists in src/
  • AGENTS.md exists in subdirectories with content
  • AGENTS.md files follow standard format
  • AGENTS.md files reference formal policy documents

1.4. License Consistency

Check V-6: Directory-License Mapping

$ ./scripts/validate-license-consistency.sh

Expected Result: Licenses match directory rules

What it checks:

  • Files in docs/ have CC-BY-SA-4.0 or correct special file license
  • Files in src/ have AGPL-3.0-or-later
  • LICENSE file has no license header (non-copyrightable)
  • Root README.md has appropriate license

1.5. HTML Validation

Check V-7: W3C Compliance

$ ./scripts/validate-html.sh

Expected Result: Valid HTML5, no errors

What it checks:

  • Valid HTML5 DOCTYPE
  • Proper DOCTYPE and charset declarations
  • Valid semantic markup
  • Embedded semantic linked data (RDFa/Microdata/JSON-LD)
  • Proper nesting and closure of tags

2. Manual Validation

2.1. Pre-Commit Checklist

Documentation Files:

  • File placed in correct location under docs/
  • CC BY-SA v4.0 license header present
  • Valid HTML5 markup
  • Semantic linked data included
  • PROV-O metadata for agent-generated content
  • Unicode NFC normalized
  • References to formal policy included if relevant

Software Files:

  • File placed in correct location under src/
  • AGPL v3.0 license header present
  • Code compiles without errors
  • KeY verification attempted (if applicable)
  • PROV-O metadata in comments for agent-generated code
  • Follows code quality standards

Root Files:

  • File is a GitHub special file or tooling configuration
  • Not an arbitrary LLM-generated document
  • License header present if copyrightable

2.2. Ambiguity Resolution

When classification is ambiguous:

  1. Consult Formal Document Policy
  2. Apply classification predicates:
    • Documentation(f) ⇔ (Location(f) = docs/*) ∧ (Purpose(f) = Exposition)
    • Software(f) ⇔ (Location(f) = src/*) ∧ (Format(f) ∈ {Java, JAR, Class})
  3. Check format: Code vs prose?
  4. Check purpose: Explanatory vs executable?
  5. Document decision rationale in AGENTS.md

2.3. Review Process

For PR Reviewers:

  1. Verify all automated checks pass
  2. Review file placement against formal model
  3. Check license headers are correct
  4. Verify AGENTS.md files updated if directories modified
  5. Confirm no arbitrary LLM files at root
  6. Validate semantic markup in HTML documentation
  7. Check PROV-O metadata for agent-generated content

3. CI/CD Integration

3.1. Pre-commit Hooks

The following pre-commit hooks should be configured in .pre-commit-config.yaml: 

repos:
  - repo: https://github.com/fsfe/reuse-tool
    rev: v3.0.2
    hooks:
      - id: reuse
        name: REUSE compliance check

  - repo: local
    hooks:
      - id: validate-file-placement
        name: Validate file placement
        entry: ./scripts/validate-file-placement.sh
        language: script

      - id: validate-agents-md
        name: Validate AGENTS.md files
        entry: ./scripts/validate-agents-md.sh
        language: script

      - id: validate-html
        name: Validate HTML5
        entry: ./scripts/validate-html.sh
        language: script

3.2. GitHub Actions Workflow

Example GitHub Actions workflow (.github/workflows/validate.yml): 

name: Validate Document Policy

on: [push, pull_request]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install REUSE tool
        run: pip install reuse

      - name: REUSE compliance check
        run: reuse lint

      - name: Validate file placement
        run: ./scripts/validate-file-placement.sh

      - name: Validate AGENTS.md files
        run: ./scripts/validate-agents-md.sh

      - name: Validate HTML5
        run: ./scripts/validate-html.sh
