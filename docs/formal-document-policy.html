Abstract

This document provides a formal, rigorous, and mathematically precise model for classifying repository artifacts into copyrightable and non-copyrightable categories, with explicit separation between works licensed under CC BY-SA v4.0 (documentation) and AGPL v3.0 (software). The model is constructed using category-theoretic methods consistent with the repository's foundational approach to subclassical sequent calculi.

Presumption of Inconsistency: As per repository policy, all classifications must be locally provable and globally consistent. Ambiguity is treated as a formal refutation requiring model revision.

1. Formal Definitions

1.1. The Repository as a Category

Let Rep be the category whose objects are file system paths in the repository and whose morphisms are inclusion relations between directories. 

Ob(Rep) = {p | p is a valid path in the repository}
Hom(Rep)(p₁, p₂) = {⊆ | p₁ is a subdirectory of p₂}

The terminal object 1 ∈ Ob(Rep) is the root directory "/".

1.2. The File Type Functor

Let Type: Rep → Set be a functor mapping each file path to its type classification: 

Type: Rep → Set
Type(p) ∈ {Documentation, Software, Metadata, Special, NonCopyrightable}

1.3. The Copyrightability Predicate

Let Copyable: Rep → {⊤, ⊥} be a predicate indicating copyrightability under international copyright law (Berne Convention, WIPO Copyright Treaty). 

Copyable(p) = ⊤  ⇔  p is subject to copyright protection

Non-copyrightable works (as established by judicial precedent in multiple jurisdictions):

  • Purely functional code lacking creative expression (threshold of originality)
  • Standardized file formats and protocols (RFCs, ISO standards text)
  • Automatically generated boilerplate (within established limits)
  • Mere facts and data compilations without creative selection/arrangement
  • Copyrighted text when reproduced verbatim in licenses or attribution notices

2. Copyrightable Works Model

2.1. Classification Schema

The repository artifact space is partitioned as follows: 

Artifacts = A_doc ∪ A_soft ∪ A_meta ∪ A_spec ∪ A_nc

where all sets are pairwise disjoint:

∀i,j ∈ {doc,soft,meta,spec,nc}, i≠j: A_i ∩ A_j = ∅
Class Set License Copyrightable Examples
Documentation A_doc CC BY-SA v4.0 ⊤ Thesis text, explanatory HTML, research prose
Software A_soft AGPL v3.0 ⊤ Java source, compiled classes, scripts
Metadata A_meta CC BY-SA v4.0 ⊤ Citation metadata, PROV-O records
Special Files A_spec Varies by tool ⊤ README.md, LICENSE, pom.xml
Non-Copyrightable A_nc N/A ⊥ License texts, RFC reproductions

2.2. The Documentation Predicate

A file f is classified as Documentation iff: 

Documentation(f) ⇔ (Location(f) = docs/*) ∧ (Format(f) ∈ {HTML, XHTML, MD}) ∧ (Purpose(f) = Exposition)

where Purpose is derived from the intent of the file:

  • Exposition: Explains theory, methodology, or design decisions
  • Specification: Defines formal properties, invariants, or contracts
  • Instruction: Provides usage guidance or operational procedures
Crucial Distinction: Documentation in the docs/ directory is the source of truth for CC BY-SA v4.0 licensed works. Generated artifacts derived from these works (e.g., compiled Java records via annotation processing) inherit their license from the derivation rules, not the source.

2.3. The Software Predicate

A file f is classified as Software iff: 

Software(f) ⇔ (Location(f) = src/*) ∧ (Format(f) ∈ {Java, JAR, Class})

The src/ directory constitutes the software implementation, which under the REUSE specification and repository policy, is licensed under AGPL v3.0.

3. Documentation Hierarchy

3.1. Canonical Documentation Structure

The documentation tree T_doc is a rooted tree with structure: 

T_doc = (V, E)
V = {d ∈ paths | d starts with docs/}
E = {(d₁, d₂) | d₂ is immediate child of d₁ in filesystem}

Each node d ∈ V must contain exactly one AGENTS.md (policy) and optionally one README.md (exposition).

3.2. Documentation Inclusion Principle

Principle: With the sole exception of GitHub special files, ALL documentation must reside under the docs/ directory.

∀f ∈ A_doc: (f ∉ Special_GitHub) → Location(f) = docs/*

This principle ensures a clean separation between user-facing documentation and implementation artifacts.

3.3. AGENTS.md vs README.md

The two standard documentation files have distinct purposes:

Property AGENTS.md README.md
Target Audience Machine Agents (LLMs) Humans
Purpose Policy, constraints, procedures Exposition, context, motivation
Format Strict, structured, machine-parseable Narrative, explanatory, flexible
Required ⊤ (every directory) ⊥ (optional)
License CC BY-SA v4.0 (in docs/) CC BY-SA v4.0 (in docs/) or AGPL v3.0 (in src/)

4. Special Files Model

4.1. Definition of "Special Files"

Special Files are files with semantic meaning to GitHub, build tools, or licensing frameworks that must reside in specific locations to function correctly. 

Special(f) ⇔ (f ∈ GitHub_Special ∪ Tooling_Special ∪ REUSE_Special)

4.1.1. GitHub Special Files

Files that GitHub recognizes and renders specially:

  • README.md — Repository landing page (root only, exempt from docs/ rule)
  • LICENSE — License display and SPDX detection (root required)
  • CONTRIBUTING.md — PR contribution guidance
  • CHANGELOG.md — Version history (SemVer compliance)
  • CODE_OF_CONDUCT.md — Community standards
  • AUTHORS.md — Contributor attribution
  • AGENTS.md — Agent policies (repository-specific special file)
  • .gitignore — Git exclusion patterns (version control metadata)

4.1.2. Tooling Special Files

Files required by build systems, frameworks, or development tools:

  • pom.xml — Maven project descriptor (AGPL v3.0)
  • build.gradle / build.gradle.kts — Gradle build scripts (AGPL v3.0)
  • .pre-commit-config.yaml — Pre-commit framework configuration (tool metadata)
  • codemeta.json — CodeMeta citation metadata (CC BY-SA v4.0)

4.1.3. REUSE Specification Special Files

Files required for REUSE.software compliance:

  • .reuse/dep5 — Debian copyright format file (license metadata)
  • REUSE compliant headers in all copyrightable files (license metadata)

4.2. The Special File Location Theorem

Theorem: Special files may exist at root level iff required by the tooling system or GitHub platform, notwithstanding the documentation inclusion principle.

∀f ∈ Special: (RequiredAtRoot(f)) → (Location(f) = /)

Proof: By definition of special files as platform/tool requirements. ∎

4.3. LLM-Generated Files Constraint

Constraint: No arbitrary LLM-generated files are permitted in the repository root.

Formal Statement:

∀f ∈ A_doc: (Location(f) = /) → (f ∈ Special_GitHub ∪ f ∈ Special_Tooling)

Rationale: Prevents repository pollution by unstructured, non-canonical AI outputs. All LLM-generated documentation must be curated into docs/ with proper validation.

5. Licensing Algebra

5.1. The License Functor

Let Lic: Artifacts → Licenses be a functor assigning licenses to artifacts: 

Lic: A_doc ∪ A_soft ∪ A_meta ∪ A_spec ∪ A_nc → {CC_BY_SA_4, AGPL_3, NONE}

5.2. License Assignment Rules

Rule Condition License Assignment
DR-1 f ∈ A_doc Lic(f) = CC_BY_SA_4
DR-2 f ∈ A_soft Lic(f) = AGPL_3
DR-3 f ∈ A_meta Lic(f) = CC_BY_SA_4 (metadata inherits from documentation)
DR-4 f = LICENSE Lic(f) = NONE (license text is non-copyrightable)
DR-5 f = README.md (root) Lic(f) = AGPL_3 (special case: repository metadata)
DR-6 f ∈ AGENTS.md Lic(f) = CC_BY_SA_4 (documentation policies)
DR-7 f ∈ REUSE_Headers Lic(f) = parent license (inherited)

5.3. The Dual Licensing Model

The repository implements a transformation functor T that maps CC BY-SA v4.0 documentation to AGPL v3.0 software: 

T: A_doc → A_soft
∀f ∈ A_doc: (T(f) ∈ A_soft) ∧ (Lic(T(f)) = AGPL_3)

Example: Semantic HTML documentation in docs/ describing Java Records is transformed via annotation processing into AGPL v3.0 licensed Java source code in src/.

Important: This transformation is consistent with the REUSE specification's allowance for dual licensing through explicit documentation of the transformation relationship.

6.2. Consistency Checking

For any artifact f, the following consistency checks must hold: 

Consistent(f) ⇔ (Copyable(f) → Lic(f) ≠ NONE)
Consistent(f) ⇔ (¬Copyable(f) → Lic(f) = NONE)

These checks ensure that copyrightable works have licenses and non-copyrightable works do not claim copyright protection.
